# State of the app ‚Äî review, July 2026

Reviewed 2026-07-24 on `docs/ci-conventions` @ `3875a80`, with the splash work
uncommitted in the tree. Baseline measured: **1365 passed, 1 skipped in 8.6 s**,
ruff clean, **79 %** line coverage overall.

This is the third pass over the codebase ([`audit-plan.md`](audit-plan.md) complete
2026-07-10, [`audit-2-plan.md`](audit-2-plan.md) complete 2026-07-21). Both of those
were structural and both landed, so this one deliberately does *not* re-tread the
layering. It looks for the two things a structural audit doesn't catch: **runtime
behaviour when the user does two things at once** (object lifetime, which the
architecture tests can't see because they check imports), and **whether two surfaces
actually agree about a rule** (which no test currently relates at all).

Every finding below names the probe that produced it. Two are reproducible process
crashes; one is a wrong number the app is shipping today.

---

## Verdict

The engine is in genuinely good shape and I want to be specific about why, because
it is the reason the findings below are as narrow as they are.

The Qt/pure split is real, not aspirational: 6,390 statements, and the entire domain
layer ‚Äî `char_rules` 98 %, `character` 96 %, `monster_parser` 99 %, `navigation`
100 %, `roll20_export` 98 % ‚Äî is testable without a display. That is what lets 1,365
tests run in 8.6 seconds, and the fast suite is what makes the rest of the discipline
affordable. `tests/test_architecture.py` turning the prose rules into checks is the
single highest-leverage thing in the repo; the CI notes and the module docstrings
explain *why* rather than *what*, which is rare and worth keeping.

The findings cluster into two gaps, each one the shadow of a strength:

> **1. The pure layer is modelled; the Qt layer's *state* is not.**
> `navigation.History` is a tested state machine. But *which* `History` ‚Äî which
> `TabContext`, which worker, which in-flight question ‚Äî is decided by ad-hoc
> attribute assignment in `MainWindow`, and nothing models or tests that. Every
> confirmed crash here is an identity or lifetime error in that unmodelled layer.
>
> **2. `char_rules` is the single source of truth; nothing tests that it is used.**
> `test_architecture.py` checks *imports* ‚Äî that PyQt stays put, that SQL stays in
> `db.py`, that no logic module imports a view. It cannot check that a module which
> already imports `char_rules` actually gets its numbers from there. So a hand-typed
> rules table in a view passes every check, and one of them has already drifted.

That is a compliment to the design, not a hedge ‚Äî these bugs have nowhere else to
live. Both gaps are closable with tests of the kind already in the repo.

## Findings ‚Äî the Qt shell and its seams

Ranked by user impact. 1 and 2 are crashes with attached repros; 3 and 4 share their

### 1. Dragging a tab desynchronises every per-tab state ‚Äî **confirmed**

[`app.py:847`](../app.py#L847) enables `setMovable(True)`. `self._tabs` is a plain
list indexed by `currentIndex()` in five places
([`app.py:651-663`](../app.py#L651-L663), [`app.py:1920`](../app.py#L1920)) and
nothing is connected to `tabBar().tabMoved`. Qt reorders its own tabs on a drag; the
parallel list does not move.

```
before any drag              currentIndex=2  widget=TabC  _tabs[2]=TabC  in sync: True
after dragging TabC first    currentIndex=0  widget=TabC  _tabs[0]=TabA  in sync: False
```

After one drag, `self.content`, `self._nav` and `self.current_page_url` all resolve
against the wrong tab. Back/forward rewrites a different tab than the one on screen,
`current_page_url` bookmarks the wrong page, and `_close_tab` pops the wrong
`TabContext` ‚Äî leaking one view and orphaning another. Nothing errors; the app
quietly operates on the wrong document.

**Fix.** Stop maintaining a parallel list. Hang the `TabContext` on the view itself
(`view._ctx`, or a `dict` keyed by the widget) and resolve through
`self._content_tabs.currentWidget()`. That makes the desync unrepresentable rather
than merely handled ‚Äî a `tabMoved` handler that reorders the list would work, but it
leaves the invariant to be re-broken by the next feature. The three properties are
the only places that need to change.

**Test.** Bind the properties to a `SimpleNamespace` with a real `QTabWidget`, call
`tabBar().moveTab(2, 0)`, assert `self.content is currentWidget()` ‚Äî the house style
from `test_charactermancer.py`, no window required.

### 2. A second search crashes the app; a slow one clobbers a fast one ‚Äî **confirmed**

[`app.py:1951-1957`](../app.py#L1951-L1957):

```python
if self._search_worker and self._search_worker.isRunning():
    self._search_worker.quit()

self._search_worker = SearchWorker(str(DB_PATH), query)
```

Two independent defects in four lines.

**`quit()` does nothing here.** It asks a thread's *event loop* to exit.
`SearchWorker.run()` ([`app.py:364`](../app.py#L364)) is an override that runs a
blocking FTS query and never calls `exec_()`, so there is no event loop to ask. The
thread runs to completion and emits `results_ready` ‚Äî still connected to
`_show_results`.

**The rebind destroys a running thread.** The assignment drops the last Python
reference to a `QThread` that is still executing. sip owns it (no parent), so the C++
object is deleted underneath the running thread.

Both halves, run against this repo's PyQt5:

```
--- drop the reference (what app.py does) ---
after quit(): still running = True
exit: -1073740791                 # 0xC0000409, Windows fail-fast abort

--- keep the reference ---
after quit(): still running = True
signals delivered: ['NEW', 'OLD']
```

The first run dies before the next statement. The second ‚Äî identical but for holding
the reference ‚Äî shows the race the crash was hiding: **the older search's results
arrive last and overwrite the newer ones.** Type "dragon", then "fireball" before the
first returns, and the results list ends up showing dragons under a fireball query.

This is not an exotic path. It is typing a second search on a 74 MB FTS index.

**Fix.** Two changes, both small:

- Keep the worker alive until it finishes ‚Äî connect `finished` ‚Üí `deleteLater` and
  clear the attribute in a `finished` slot, exactly as `_ask_question` /
  `_ask_worker_done` already do for `AskWorker` ([`app.py:1738-1742`](../app.py#L1738-L1742)).
  That pattern is already correct in this file; the search path just predates it.
- Stamp each search with a monotonically increasing id and have `_show_results`
  ignore anything but the newest. `quit()` cannot cancel a blocking query, so
  discarding the stale answer is the only real fix ‚Äî and the decision *"is this
  result still wanted?"* is pure, so it belongs in a tested module rather than in the
  slot.

### 3. Asking Jarvis a second question hits the same crash ‚Äî same root cause

[`app.py:1716`](../app.py#L1716) `_ask_question` never calls `_ask_stop()`. Only
`_render_ask` does. And the `loading` state still renders the ask bar
([`askscreen_html.py:254`](../askscreen_html.py#L254) ‚Äî `_askbar()` is present and
not disabled), so submitting a second question while the first streams is a
supported gesture in the UI.

Doing it rebinds `self._ask_worker` over a running `AskWorker`: the crash from
finding 2, with a longer window to hit it because a local LLM takes seconds, not
milliseconds. Even surviving that, `self._ask.begin()` overwrites `question` and
`view` before the first worker returns, so the first answer gets recorded against the
second question ‚Äî `record_answer` reads whatever `question` now holds.

**Fix.** `self._ask_stop()` as the first line of `_ask_question`, and let
`Conversation` refuse to `begin` while a question is in flight ‚Äî that decision is
pure and `ask_controller` is already the tested home for it.

### 4. Mini-app state is window-level while history is per-tab

`self._cm`, `self._mon`, `self._mon_saved_id` and `self._ask` live on `MainWindow`
([`app.py:615-627`](../app.py#L615-L627)) while `History` lives on `TabContext`. So
the builder is *addressable* per tab ‚Äî `"charactermancer"` is a destination, Back
works, a second tab can navigate to it ‚Äî but there is only one build behind every
such tab.

Open the builder in two tabs and they render the same character; act in one and the
other's DOM is now stale, with no re-render on tab switch. Same for the monster sheet
and the Jarvis thread. It is not a crash and it is arguably fine for a single-user DM
tool, but it is the one place where the tab model promises something the state model
does not deliver, and it is the reason findings 1 and 3 were reachable.

**Fix, when it's worth it.** Move the three onto `TabContext`. It composes with
finding 1's fix ‚Äî once context is resolved from the widget rather than an index,
adding fields to it is free. Not urgent; worth doing *with* finding 1 rather than
after, since it touches the same lines.

### 5. `calculator.py` is 12 % covered, and the untested part is a rule

The module is 148 lines of Qt widget construction plus `_update_hit`
([`calculator.py:131-148`](../calculator.py#L131-L148)), which is not construction ‚Äî
it decides the phrasing for three distinct combat outcomes ("Hits on 2+ ‚Ä¶ only a
natural 1 misses", "Can't hit by the numbers", "Need N+ ‚Ä¶ crit on 18+") and computes
the crit threshold by searching the rule rather than hard-coding it. That last part
is careful work, and it is the least-tested code in the app.

The docstring's claim that the module is "pure display on top" is *nearly* true ‚Äî
and the gap is exactly the interesting bit. Everything it needs is already imported
from `char_rules`.

**Fix.** Extract `to_hit_summary(bonus, target) -> str` into `char_rules` beside
`to_hit_need` / `is_critical`, leave `_update_hit` as `setText(...)`, and table-test
the boundaries: `need <= 1`, `need > 20`, `crit_from == 20`, `crit_from is None`.
Roughly twenty lines of test for the app's most-consulted number.

### 6. The theme migration is no longer blocked ‚Äî the splash just proved it

[`theme.py`](../theme.py) documents why the CSS blocks still hold hex literals (plain
triple-quoted strings; `{TEXT}` would need every brace in the stylesheet escaped) and
names the safe route: emit `:root { --text: ‚Ä¶ }` and have the CSS say `var(--text)`,
because `var(--x)` survives a non-f-string untouched.

Current count across the non-generated modules: **247 distinct hex colours**, with
`#c9a84c` ‚Äî `theme.ACCENT` ‚Äî typed out 37 separate times. Only six modules import
`theme` at all; `charactermancer_html.py` carries ~353 CSS lines and imports none of it.

What changed: the uncommitted `splash_html.py` already does this. It declares
`--blue` / `--red` on `body` and uses `var(--blue)` throughout, plus per-element
`style="--c:{color}"` for the tool tiles ‚Äî and it renders correctly in this
QtWebEngine. The "large edit with no way to verify" caveat that deferred audit-2's
finding 5 has been answered by a screen that shipped.

**Fix.** One `theme.css_vars()` returning a `:root` block, prepended per screen,
migrated a screen at a time ‚Äî `monster_html` and `charactermancer_html` first, since
they are the two biggest CSS blocks. Mechanical, reversible, one screen per commit.

### 7. The new splash generator should join the CI drift gate

`scripts/build_splash_assets.py` is stdlib-only (`base64`, `pathlib`), deterministic,
and reads two committed inputs. That is precisely the profile the
"Generated modules match their sources" step in
[`tests.yml`](../.github/workflows/tests.yml) was built for ‚Äî unlike `build_spells.py`
and `build_toc_tree.py`, it does not touch the network, so the reason those two are
excluded doesn't apply.

Without it, replacing `assets/adnd2e-logo.png` and forgetting to regenerate leaves
the shipped splash showing the old art with nothing to catch it. Adding
`python scripts/build_splash_assets.py` and extending the `git diff --exit-code`
paths is a two-line change.

Two notes on the same work, neither blocking:

- **`splash_assets.py` is 96 KB of base64 in source.** Fine given a 74 MB DB is
  already committed, and it does buy a real property (no runtime file access, so dev
  and frozen bundle behave identically). Worth stating in the docstring as the
  deliberate trade it is ‚Äî the alternative, shipping the `.otf` via PyInstaller
  `datas` and reading from `_bundle_dir()`, is what `_app_icon()` already does, so a
  future reader will reasonably ask why the two differ.
- **The font is personal-use-only.** `assets/DeathStar-readme.txt`: *"free for
  personal use only as long as this readme file stays intact. For commercial use
  please contact me."* The generator's docstring already records this, which is good
  practice. The thing to be aware of is that `build.yml` now publishes tagged GitHub
  releases, so the font ships inside a public artifact ‚Äî a different act from using
  it locally. Non-commercial, so plausibly within the licence; just worth being a
  decision rather than a side effect of adding a release job.

## Findings ‚Äî the feature code

Different character to the section above. The shell's bugs were about *lifetime*;
these are about **whether two surfaces agree about a rule**. That distinction matters
because the domain layer's whole design premise is that they can't.

First, the part that holds up, since it sets the bar for what follows. I spot-checked
the transcriptions in `char_rules` against the printed tables: Strength (incl. all
five exceptional bands), Dexterity, Constitution, Intelligence, Wisdom and the
cumulative priest bonus spells, plus the wizard/priest/paladin/ranger/bard spell
progressions (Tables 21, 24, 17, 18, 32) and Table 60's saves. **Every value checked
was correct.** The footnote handling is right too ‚Äî priest 6th-level spells gated on
Wis 17+ and 7th on Wis 18+, bonus spells applied only to levels the priest can
actually cast, paladins and rangers correctly excluded from Wisdom bonuses. The
`test_char_rules` assertion that Table 61 is a perfect diagonal is a genuinely smart
way to catch a transcription typo, and it works.

The layering holds in the feature code as well: `monster_html` reaches char_rules
only through `monster`, `roll20_export` re-exports rather than reimplements, and
`damage_to_dice` serves both the sheet's terse rendering and Roll20's canonical one
from a single rule. `Character.from_dict`'s three migrations (flat weapon-prof list ‚Üí
mastery dict, flat spell list ‚Üí level dict, `bought_ambidexterity` ‚Üí the CT talent)
are exactly the right shape.

Which is why the following is worth taking seriously.

### 9. The DM Screen's THAC0 card disagrees with the rules engine ‚Äî **confirmed, shipping**

[`dmscreen_html.py:218-229`](../dmscreen_html.py#L218-L229) hard-codes four THAC0
columns as a literal. `char_rules._thac0_raw` derives the same numbers, with a
docstring reading *"verified against the PHB"*. They do not match.

```
column        vs RAW    vs builder (house rules on)
Warrior         0/20                           0/20
Priest          6/20                           6/20
Rogue           0/20                          15/20
Wizard          0/20                           0/20
```

Two separate defects:

**The Priest column is a transcription error.** Priests improve 2 points every 3
levels, so the column is flat within each band ‚Äî 20/20/20, 18/18/18, 16/16/16. The
card instead smooths the steps, and is wrong at levels **3, 6, 9, 12, 15 and 18**. A
9th-level cleric reads THAC0 15 on the screen and 16 in the builder. That is wrong
against the PHB *and* against the app's own engine; there is no reading under which
the card is right.

**The Rogue column shows rules this campaign doesn't use.** `HOUSE_RULES.rogue_attack_as_priest`
is on, and `cr.thac0` applies it everywhere else in the app ‚Äî so the builder gives a
thief the priest progression. The card shows RAW. They differ at **15 of 20 levels**,
by up to 3 points at 19th‚Äì20th:

```
 Lvl  DM Screen  builder(house)   raw
  10         16              14    16
  19         11               8    11
```

This is the DM's quick-reference card. It is the one surface in the app whose entire
purpose is to be read mid-combat and trusted without cross-checking.

**What makes it a design finding rather than a typo:** `dmscreen_html.py` already
imports `char_rules` ‚Äî for exactly one table, the size/initiative rows at
[line 161](../dmscreen_html.py#L161), under a comment noting that sourcing it from
char_rules is what keeps the DM Screen and the monster sheet from disagreeing. The
right instinct is in the file, applied once, and not carried through to the four
tables where it would have mattered most.

**Fix.** Build the THAC0 card from `cr.thac0(class_name, level)` ‚Äî twelve lines
become a comprehension, the house rule is picked up for free, and the card can no
longer drift from the builder. Then do the same for the saves and turn-undead cards
(see finding 10), and pin it with a test that walks every level of every column
against char_rules ‚Äî the check that produced this finding, kept.

### 11. `Character.to_dict` is hand-written, and nothing pins it to the dataclass

[`character.py:829-852`](../character.py#L829-L852) lists 29 fields by hand.
`Monster.to_dict` is `asdict(self)` and cannot fall behind its dataclass; `Character`
can, and silently ‚Äî a field added to the dataclass but missed here never reaches the
save blob, `from_dict` fills it with the default, and the user's choice is quietly
gone on reload.

The existing round-trip tests wouldn't catch it: each asserts specific fields survive,
so a *new* field is exactly the thing they don't cover. `to_dict` is currently complete
(I diffed it against the dataclass ‚Äî only `rolled_pool` is absent, deliberately, and
`from_dict` excludes it symmetrically).

**Fix.** One assertion, in the spirit of `test_blob_store.py` pinning the generated
DDL:

```python
assert set(Character().to_dict()) == {f.name for f in fields(Character)} - {"rolled_pool"}
```

Switching to `asdict` would work too, but the hand-written version exists to *exclude*
`rolled_pool` and to copy the mutable containers defensively ‚Äî so pin it rather than
replace it.

### 12. Two small accuracy slips in the AC code

Minor, but this codebase is one where the docstrings are load-bearing:

- [`character.py:784`](../character.py#L784) ‚Äî `armor_class()` documents itself as
  *"(None until Dex is set)"*. It never returns None: `cr.armor_class` treats a
  missing Dex as a 0 adjustment and returns `10 + worn`. `ac_components` two methods
  below documents that behaviour correctly, so the file contradicts itself.
- [`char_rules.py:1848`](../char_rules.py#L1848) ‚Äî `armor_class(worn, dex, house_rules)`
  never reads `house_rules`. Either the ascending-AC base is house-ruled (in which
  case RAW should return a descending value) or the parameter should go. Callers pass
  it, so it reads as a supported switch that does nothing.

### 13. The Bird page loses every creature's attacks and damage ‚Äî **confirmed**

`_COMPACT_HEADERS` ([`monster_parser.py:765`](../monster_parser.py#L765)) maps the
summary grids' abbreviated column headers onto Monster fields. It knows
`# OF ATT`, `# ATT`, `NO ATT` and `DMG/ATT`, `DAMAGE/ATTACK`, `DMG`, `ATTACK`.

The Bird table writes them **singular**:

```
Bird   (MM/DD03810.htm) header: ['#AP','AC','MV','HD','THAC0','# AT','DMG/AT','MORALE','XP VALUE']
   UNMAPPED -> ['# AT', 'DMG/AT']
Insect (MM/DD03958.htm) header: [... '# OF ATT','DMG/ATT' ...]   UNMAPPED -> []
Mammal, Fish                                                      UNMAPPED -> []
```

So all **18** creatures on that page ‚Äî Blood Hawk, Giant Eagle, Boobrie, Eblis,
Falcon ‚Ä¶ ‚Äî import with no attacks and no damage, and export to Roll20 with **zero
weapon rows**. That is the entire content of a monster sheet for a DM about to run
one. It is also exactly the 18 that showed up as the corpus-wide count of creatures
missing `damage_attack`, so the Bird page is the whole of that gap.

**Fix** ‚Äî two keys, verified:

```python
"# AT": "no_of_attacks",   "DMG/AT": "damage_attack",
```

```
BEFORE: 18 birds, 18 with no damage/attacks
    Blood Hawk    #AT=''   DMG=''            roll20 weapons=0
AFTER:  18 birds,  0 with no damage/attacks
    Blood Hawk    #AT='3'  DMG='1-4/1-4/1-6' roll20 weapons=3
corpus-wide creatures with no damage_attack: 18 -> 0
```

The deeper point: the map is a hand-maintained list of abbreviations, and there is no
check that every compact grid's headers are fully consumed. A test asserting
*"across all MM pages, every compact-table header cell maps to a field"* would have
caught this the day the page was first parsed, and would catch the next spelling.

### 14. The MM's `(x N)` multiplier becomes the weapon's *name* in the Roll20 export

`_monster_weapons` ([`roll20_export.py:263`](../roll20_export.py#L263)) splits the
damage line on `/` and calls `_paren_label` to name each attack ‚Äî right for
`1-6 (claw)`, wrong for the MM's other use of parentheses, which is "this attack, N
times":

```
Aboleth          '1-6 (x 4)'                -> weapon names ['x 4']
Kraken           '3-18(x2)/2-12(x6)/7-28'   -> weapon names ['x2', 'x6', 'Attack 3']
Octopus, Giant   '1-4 (x6)/2-12'            -> weapon names ['x6', 'Attack 2']
Squid, giant     '1-6 (x8)/5-20'            -> weapon names ['x8', 'Attack 2']
```

**12 creatures**, and they are not obscure ones. The Kraken arrives in Roll20 with
attacks called "x2" and "x6", and the count itself ‚Äî 2 tentacles, 6 arms ‚Äî is thrown
away rather than becoming 2 and 6 rows.

**Fix.** Recognise `(x N)` in `_paren_label` as a multiplier rather than a label:
emit N rows (or one row named "Attack √óN"), and fall through to the existing label
behaviour for everything else. The regex is `\(\s*x\s*(\d+)\s*\)`, case-insensitive ‚Äî
that is the form all 12 use.

### 15. 92 creatures have no Size ‚Äî that one is the book, not the parser

Size is empty on 92 of 634 creatures (14.5 %), and 90 of those come from the five
compact grids. I checked whether the header map was at fault: it is not. The grids
carry only `#AP, AC, MV, HD, THAC0, #AT, DMG/AT, MORALE, XP VALUE` ‚Äî the MM simply
does not print Size, Alignment, Intelligence, Treasure or Frequency in its summary
tables. The parser is faithfully representing what the book contains, and the ~15 %
empty rate across those five fields is the source data.

Worth recording because the *consequence* is silent. `Monster.initiative_modifier()`
derives from Size, so those 92 creatures show **no initiative speed factor** on the
sheet, and `_monster_weapons` sends `speed: 0` to Roll20. Nothing says why.

**Fix** (small, UX rather than parsing). `Monster.initiative_override` already exists
and the sheet already lets the DM set it ‚Äî so the sheet should render the field as
"not given in the MM ‚Äî set it" rather than blank, turning a silent gap into a
one-click fix. No new data required.

## How the books are stored and displayed

Checked against the real `dnd2e.db` (4,609 pages, 74 MB).

**Storage is clean, and the audit-2 fix genuinely closed the drift it was aimed at.**
All ten books in `pages` match `theme.BOOKS` exactly ‚Äî same codes, byte-identical
names, one `book_code` ‚Üí one `book_name`, no book in the theme without pages, none in
the DB missing from `BOOK_ORDER`:

```
AEG 182  CT 492  DMG 965  ECO 17  HLC 256  MM 330  PHB 962  SM 425  SP 512  TM 468
                                                                       all ok
```

**Performance is fine and needs nothing.** `pages` carries only the UNIQUE index on
`page_url` and `toc_entries` has no index at all, so I measured the display path
rather than assuming: `chapter_keyword_for_page` ‚Äî which runs on *every* page render
‚Äî is 0.29 ms, a book's TOC 0.66 ms, and the full `SCAN pages` behind the monster
import 9.8 ms. At this corpus size the indexes would buy nothing. Noting it so the
next person doesn't add them speculatively.

**The `toc_tree` fallback is broader than its docstring claims ‚Äî luckily.**
`db.toc_tree`'s docstring says the `[]` return is for "an older DB with no toc_tree
table". In fact `_load_topics` branches on `if tree:`, so it also catches *a book with
no rows in the tree* ‚Äî which is real: **ECO has 0 toc_tree rows**, falls through to
the flat `toc_entries` layout, and renders all 17 pages across its 17 chapters
correctly. Working behaviour resting on an accident; the docstring should say so.

### 16. 25 pages exist in the DB but cannot be reached by browsing

For the books that *do* have tree rows, `_load_topics` takes the tree branch and never
consults `toc_entries` ‚Äî so any page missing from `toc_tree` is unreachable from the
sidebar:

```
book   pages  in tree  unreachable
SP       512      494           18
HLC      256      254            2
TM       468      466            2
AEG      182      181            1
DMG      965      964            1
SM       425      424            1
```

These are real content pages, not artefacts:

```
SP/DD02868  Strength (Skills and Powers)          SP/DD02880  Willpower (Skills and Powers)
SP/DD02873  Reason (Skills and Powers)            AEG/DD00116 Polearms
SP/DD02875  Knowledge (Skills and Powers)         TM/DD04161  Wizard Spells-- Chapter 2
SP/DD02878  Intuition (Skills and Powers)         SM/DD03420  Spheres of Access
```

The SP block is the *Skills & Powers ability-score pages* ‚Äî the material this
campaign's house rules are built on, including the Willpower page whose name the
builder already borrows for Wisdom. `TM/DD04161` is a chapter opener.

Second-order effect: `_book_page_order` is built from the same tree walk, so these
pages are absent from it, and `_adjacent_page` returns None for a url not in the
order. A reader who arrives at one via search or a cross-reference finds **Prev and
Next both dead** ‚Äî stranded on a page with no way to walk out of it.

**Fix, and it needs no new data:** all **25 of 25** are already present in
`toc_entries`. Union the two sources ‚Äî render the tree, then append any
`toc_entries` page the tree didn't cover under a trailing "More" node ‚Äî and add them
to `_book_page_order` either way, so Prev/Next works even if the sidebar placement is
imperfect. The check that found this (per book: `pages` ‚àí `toc_tree`) is worth keeping
as a test with the count pinned at 0.

## UX

Small, evidence-backed, in rough value order.

- **Search can't be scoped to a book.** `db.search_pages` has no `book_code` filter,
  so "specialization" returns PHB, CT and SP rows interleaved with no way to narrow.
  For a reference app where the user usually knows which book they want, a book
  filter on the results tab is the highest-value single UI addition available. It is
  one optional SQL parameter and a row of toggles; the per-book colours to render
  them already exist in `theme.BOOKS`.
- **The browse pane's visibility is the one workspace state not remembered.**
  `_restore_session` force-hides it unconditionally
  ([`app.py:2074`](../app.py#L2074)) while tabs, zoom, geometry and the calculator's
  position all persist. The comment explains the *mechanism* (QSplitter re-shows its
  panes on first display) but the fix has become the policy. Save the flag and
  restore it in the same `singleShot`.
- **Verify the `[` / `]` shortcuts against the web view's text fields.** Bare
  single-key shortcuts registered on the window ([`app.py:954-955`](../app.py#L954-L955))
  are a classic Qt hazard. I confirmed `QLineEdit` is safe ‚Äî it accepts the
  `ShortcutOverride` for printable keys, so the search and find boxes swallow the
  bracket correctly. I could **not** verify `QWebEngineView`; it segfaults under
  `offscreen` in this environment. If it does not claim the key, then typing `[` into
  the Jarvis question box, the character-name field, or a monster sheet's editable
  field navigates the tab to the previous page mid-edit. Thirty seconds in the
  running app settles it; if it is broken, gate both shortcuts on
  `hasFocus()`/`focusWidget()` or move them behind a modifier.
- **Status-bar messages carry outcomes that deserve more weight** ‚Äî "Roll20 JSON
  copied", "Search failed". They persist until the next `showMessage`, so nothing is
  missed outright, but a clipboard export is a *result*, and the status bar is
  `#44485c` on `#0e0f14` at the bottom of a 1200 px window. A brief inline
  confirmation on the surface that triggered it would land better.
- **No light theme.** Correct call for the product; noting only that the dark palette
  is now hard-coded in ~247 places, so the cost of ever reconsidering is set by
  finding 6, not by the decision itself.
